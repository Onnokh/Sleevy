# Connect Flow for First-Party Clients

Sleevy will issue API Keys to first-party clients (Chrome extension, Raycast plugin, future automation) through a custom OAuth-style consent flow with PKCE rather than adopting BetterAuth's OIDC Provider plugin. A `sleevy.app/connect` consent screen, gated by the existing BetterAuth session, mints a one-time code that the client exchanges for a long-lived, scoped API Key via two new API endpoints; clients store the resulting key in their native secret storage and authenticate exactly like today's bearer-token paths. Scopes are stored on the existing `apiKey.permissions` field with a fixed v1 vocabulary, all pre-existing keys are backfilled to that vocabulary in a one-time migration, and `permissions = null` becomes deny-by-default rather than full-access. The OIDC plugin was rejected as overkill for two first-party clients with no public developer platform; the custom two-endpoint design preserves a single bearer-token model across iOS, web, and extensions, and revocation rides on the Connected Apps section of `/settings`. Design detail lives in [docs/connect-flow.md](../connect-flow.md).
