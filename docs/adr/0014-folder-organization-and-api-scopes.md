# Folder Organization and API Scopes

Sleevy will model Folders as user-owned, flat Library destinations separate from Tags, with each Saved Item assigned to at most one Folder through a nullable foreign key using `ON DELETE SET NULL`, and returning `folder` as a nullable embedded Folder identifier/name summary. Folder names preserve trimmed display casing while Postgres enforces Account uniqueness with an index on `(user_id, lower(name))`. Folder Views are loaded through `GET /v1/saved-items?folder={folderId|none}` rather than client-only filtering. The shared REST API will expose folder listing and management according to explicit `folders:read`, `folders:write`, and `folders:delete` scopes; `saved-items:capture` may assign a known Folder without `folders:read`, while `saved-items:write` may change existing Folder membership. Capture always establishes the current Folder destination: a Folder identifier files the Saved Item there and its absence files it in No Folder, including on duplicate capture. Older one-tap clients remain operational but intentionally move recaptured items to No Folder; Sleevy accepts this trade-off to keep capture semantics uniform while supporting self-contained rendering, scalable Folder navigation, least-privilege automation filing, and full clients without granting folder enumeration or management implicitly or turning Tags into personal containers.
